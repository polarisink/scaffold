# 模块分层与设计约束

## 当前结构

```text
scaffold/
├── scaffold-biz/           # 默认业务启动项目
├── scaffold-core/          # 基础实现与底层抽象
├── scaffold-starters/      # 对外暴露的自动装配入口
├── scaffold-module/        # 可选业务模块
└── scaffold-test/          # 示例与技术实验
```

## 设计边界

### core

- 不表达业务语义
- 不依赖业务包路径扫描才能生效
- 适合承载基础实体、通用 mapper、底层工具和公共配置实现

### starter

- 负责组合 `core` 与第三方依赖
- 负责默认 Bean、条件装配和配置属性绑定
- 面向业务方暴露稳定接入语义

### module

- 负责业务领域能力
- 不绕开 starter 直接拼装底层技术栈
- 需要明确接入依赖、缓存、表结构和配置项

## 默认策略

- 文件能力默认关闭
- Swagger 默认关闭
- 安全与跨域策略默认保守
- 实验模块不进入默认构建

## 演进原则

- 优先新增 starter，而不是让业务模块直接依赖杂散 core
- 能用条件装配解决的问题，不依赖启动类显式硬编码
- 每次新增基础能力时，同步补自动装配测试和文档
